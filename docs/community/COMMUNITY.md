4. Respond to review feedback

2. Ask for feedback
3. Cross-post to other platforms

## Contribution Guidelines

### Code Quality

**We care about:**
- ✅ Type safety (TypeScript strict mode)
- ✅ Accessibility (keyboard nav, ARIA labels)
- ✅ Performance (< 100ms interactions)
- ✅ Mobile responsive (works on phones)

**We don't require:**
- ❌ Perfect code (we'll help refine in PR reviews)
- ❌ Full test coverage (nice to have, not mandatory)
- ❌ Corporate-style commits (casual is fine)

**Philosophy:** Ship fast, iterate together.

---

### Communication Style

**Be:**
- Friendly (we're building something fun!)
- Direct (clear communication > politeness)
- Patient (we're all learning)

**Don't:**
- Spam (wait for responses before pinging again)
- Demand (we're volunteers, not employees)
- Gatekeep (help newcomers, don't intimidate)

---

### Decision Making

**How features get approved:**
1. Proposal in Discord or GitHub issue
2. Community feedback (thumbs up/down)
3. Maintainer review (alignment with mission)
4. Merge + deploy

**Democracy with benevolent dictatorship:** Community votes, maintainers decide.

## FAQ

### "I'm a beginner. Can I contribute?"

**Yes!** Check `docs/GOOD-FIRST-ISSUES.md` for beginner-friendly tasks. We'll mentor you through your first PR.

---

### "I don't code. Can I still help?"

**Absolutely!** We need:
- Social media sharing (Twitter, Reddit, Discord)
- Blog posts / tutorials
- Bug testing on different devices
- Documentation improvements

**Non-code contributions are equally valued.**

---

### "How fast do PRs get reviewed?"

**Goal:** 24-48 hours for first response.

**Reality:** Depends on maintainer availability. If > 3 days with no response, ping in Discord.

---

### "Can I work on something not in the roadmap?"

**Yes, but...**
1. Open a GitHub issue first (propose your idea)
2. Get feedback from maintainers
3. If approved → build it

**Why:** Prevents wasted effort on features that won't merge.

---

### "What if my PR gets rejected?"

**Happens sometimes.** Common reasons:
- Doesn't align with mission
- Performance concerns
- Breaks existing features

**Our commitment:** We'll explain why + suggest alternatives. Rejection ≠ personal.

---

### "How do I become a maintainer?"

**Path:**
1. Contribute consistently (5+ merged PRs)
2. Help other contributors (reviews, Discord support)
3. Show judgment (good decisions on features/bugs)
4. Express interest

**Timeline:** 2-3 months of active involvement.
